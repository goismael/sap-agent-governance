Core principles
01

Least-privilege by design

Agents receive only the authorization objects required for their specific function — nothing more, no exceptions.

02

Separation of duties across agents

Detection, recommendation, and execution live in different agents with different permission profiles.

03

Human-in-the-loop at consequential points

Any action that changes a posted document, triggers a reversal, or exceeds a materiality threshold requires human authorization.

04

Full auditability of agent reasoning

Every tool call, recommendation, and escalation is logged with enough context to reconstruct why the agent acted.

05

Graceful escalation over silent failure

Agents surface ambiguity clearly and immediately. They never resolve uncertainty autonomously.

P001 Agent Permission Scoping Against SAP Authorization Objects
Critical Available
▾

When an AI agent authenticates to SAP S/4HANA, it typically runs as a service user. Without deliberate scoping, that user inherits broad authorization profiles — often far beyond what the agent needs. This pattern defines how to scope agent permissions correctly against SAP authorization objects, using dedicated PFCG roles, least-privilege ACTVT assignments, and a runtime permission guard that enforces scope at execution time.

F_BKPF_BUK F_BKPF_BLA F_SKA1_BUK S_RFC S_SERVICE PFCG SU01 SU24
# Runtime permission guard — called before every SAP interaction class AgentPermissionGuard: def check_bapi_call(self, bapi, company_code, doc_type=None): if company_code not in self.scope.permitted_company_codes: self._log_and_raise([ f"Company code '{company_code}' outside agent scope" ])

P002 Human Approval Gates for High-Risk SAP Transaction Codes
Critical Coming
▾

Not all agent actions are equal. Posting a standard journal entry (SA) carries different risk than reversing a document via FB08 or applying an intercompany elimination. This pattern defines a risk classification framework for SAP transaction codes, and shows how to implement approval gates that pause agent execution and route decisions to the appropriate human authority before consequential actions are taken.

P003 Audit Logging for Agentic Workflows in SAP
High Coming
▾

Auditors reviewing an AI-driven financial close need to answer one question: why did the agent do what it did? Traditional application logs don't capture agent reasoning — only system events. This pattern defines a structured logging schema for agentic SAP workflows that captures tool calls, model reasoning, human approvals, and escalations in a format that satisfies internal audit, external audit, and SOX compliance requirements.

P004 Agent Failure Handling During Financial Period-End Close
High Coming
▾

Period-end close operates under time pressure. When an agent fails mid-workflow — due to an API timeout, a data quality issue, or an authorization error — the system must fail safely without leaving the ledger in an inconsistent state. This pattern defines checkpoint design, rollback strategies, and escalation protocols that preserve data integrity and allow clean recovery without manual re-entry of completed steps.
